### 1. The Mechanism: Z-Encoding

To avoid clashing with standard assembly or C linkages, GHC replaces any non-alphanumeric character with a `z` (or `Z` for uppercase) followed by a specific mnemonic character.

Here are the most common Z-encoding mappings you will encounter in GHC assembly:

| Original Character | Z-Encoded | Common Context |
| --- | --- | --- |
| `_` (underscore) | `zu` | Standard spacing or internal prefixes. |
| `$` (dollar) | `zd` | Used by GHC for special optimizer passes. |
| `#` (hash) | `zh` | Ubiquitous for unboxed/primitive types (e.g., `Int#`). |
| `-` (dash) | `zm` |  |
| `:` (colon) | `zc` |  |
| `'` (tick) | `zq` |  |
| `z` / `Z` | `zz` / `ZZ` | Escaping the letter 'z' itself. |

### 2. Decoding the `Wut_mine` Symbols

Using the Z-encoding table, we can reverse-engineer the massive string you found: `Wut_minezuzdsmine_closure`.

* `Wut_` is the module name.
* `mine` is your original function.

### 3. The Anatomy of an STG Symbol

In the AArch64 output, you'll also notice that a single function like `mine` doesn't just have a `_closure` suffix. GHC breaks functions down into runtime components:

* **`_closure`:** The statically allocated heap object.
* **`_info` (Info Table):** Metadata that sits directly next to the code in memory. It tells the runtime's Garbage Collector exactly how many pointers and non-pointers are in the closure's payload.
* **`_entry`:** The actual executable AArch64 machine code. *(Note: In modern GHC versions using "tables-next-to-code", the entry code and info table are often merged into a single contiguous block, where the `_info` symbol points to the bytes immediately preceding the first executable instruction).*
* **`_con_info`:** Specifically marks the info table for a data constructor.

### 4. Connection to Init Performance

Since you are trying to reproduce an init performance observation, this explosion of symbols is the exact right place to look.

Every static closure (and its associated info table) generated by GHC optimizations adds to the `.data` and `.rodata` sections of your compiled binary. When the OS loads the binary, an abundance of statically allocated closures can noticeably degrade cold-start initialization performance due to:

1. **Relocations:** The dynamic loader must resolve all pointers between these closures, info tables, and code blocks. If you are compiling with `-fPIC` for a shared library or PIE executable, this overhead scales linearly with the number of generated closures.
2. **Page Faults / Cache Pressure:** More symbols mean a physically larger memory footprint and more page faults during the application's boot sequence.

If the function `mine` is inadvertently triggering massive specialization cascades or floating out hundreds of static thunks, it will directly manifest as the init performance lag you are analyzing.


### 2. Why Functional Languages Want *All* the Registers

When you look at the `ghccc` specification in LLVM, you will notice it passes almost everything in registers and disables traditional callee-saved register preservation. There are a few major reasons why GHC (and similar functional compilers) do this:

* **The STG Abstract Machine:** GHC doesn't compile Haskell directly to hardware concepts; it compiles it to an abstract machine called the Spineless Tagless G-machine (STG). The STG has its own set of virtual registers (e.g., `BaseReg`, the Stack Pointer `Sp`, the Heap Pointer `Hp`, and general-purpose registers `R1` through `R10`). To make Haskell fast, GHC permanently pins these STG virtual registers to physical hardware registers.
* **Everything is a Tail Call:** Haskell programs are fundamentally driven by continuation-passing style and tail calls. Instead of "calling" a function and "returning" to the caller, a Haskell function typically computes a value and then makes a direct jump (tail call) to the next execution block.
* **The Death of the C Stack:** Because Haskell rarely "returns" in the standard sense, pushing and popping standard C stack frames to save/restore callee-saved registers is pure overhead. `ghccc` treats the hardware registers as global state for the STG machine. When one closure jumps to another, the arguments and machine state are already sitting in the exact registers the next closure expects.

By taking ownership of all registers, GHC entirely bypasses the C stack. It manages its own separate, garbage-collected stacks on the heap (using the pinned `Sp` register).

## 1. `Rec` and Recursive Bindings

In Core, local bindings are explicitly partitioned into mutually recursive blocks (`Rec { ... }`) and non-recursive bindings (which just look like standard `let x = ...`).

You noticed that not all recursive functions seem to have `Rec`. There are a few reasons why:

* **Worker/Wrapper Splits:** GHC often splits a recursive function into two parts. The outer function (the "wrapper") handles standard Haskell boxed types and is **not** recursive. The inner function (the "worker", usually prefixed with `$w`) takes raw unboxed types and *is* recursive. You might just be looking at the wrapper!
* **Top-Level Definitions:** At the very top level of a module, GHC sometimes omits the explicit `Rec` block in the dump for readability, even if the functions call each other. `Rec` is strictly enforced for local `let` bindings.
* **Join Points:** Sometimes recursive functions are transformed into `join rec`, which we will cover below.

## 2. `case` Drives Evaluation (Not Just Branching)

In standard Haskell, `let` means "allocate a lazy thunk." In Core, `case` is the **only** way to force evaluation.

When you see a `case` that ignores its result (e.g., `case x of _ -> ...` or `case x of { DEFAULT -> ... }`), it is not pattern matching. It is GHC saying: *"Evaluate `x` to weak head normal form right now, then continue."*

Furthermore, `if-else` does not exist in Core. It is always translated to:

```haskell
case x of {
  False -> ... ;
  True  -> ...
}

```

If you are hunting an `if-else` depth issue, look for massive, deeply nested `case` blocks.

## 3. The Magic Hash (`#`) Means Raw Metal

Because `-dsuppress-all` hides type signatures, the hash suffix is your best friend for understanding what is actually happening in memory.

* `Int` is a pointer to a boxed heap object containing an integer.
* `Int#` is a raw, unboxed machine integer (like `i64` in Rust) sitting in a hardware register.
* `+#`, `==#`, `>#` are the low-level, unboxed machine instructions for those operations.

If you see a loop using `Int` and `+`, it is allocating memory on every iteration. If it uses `Int#` and `+#`, it is running as fast as C.

## 4. Join Points (`join` and `join rec`)

You will frequently see `join j x = ... in ... jump j 5`.

This looks exactly like a local `let` function, but it is deeply different. A `join` point is a guarantee by the compiler that this "function" will **only** be tail-called. Because of this guarantee, GHC does not allocate a closure for it on the heap. Instead, it compiles it into a literal `goto` instruction (a direct jump), and the arguments are just passed in registers.

When a recursive function is highly optimized, it often becomes a `join rec` block instead of a `let Rec` block. This is a massive performance win.

## 5. Dictionary Passing (`$d` or `$c`)

Haskell's typeclasses (which are conceptually identical to Rust's Traits) do not exist at runtime. GHC implements them by passing hidden "dictionaries" — essentially structs containing function pointers.

If you have a function with an `Eq a =>` constraint, in Core you will see an extra argument usually named something like `$dEq`. If you see these being passed around inside your tight loops, it means the compiler failed to specialize the function, and you are paying the cost of indirect dynamic dispatch on every iteration.

## 6. Casts and Coercions (`cast`)

Even with suppression turned on, you might occasionally see `expression `cast` something`.
Haskell has `newtype`, which allows you to create a new type that is physically identical to an old type at runtime (like `struct Wrapper(i32);` in Rust). In Core, GHC uses `cast` to explicitly prove to the type checker that it is converting between these types, but know that a `cast` has **zero runtime cost**. It emits no machine code.

Understanding how `Rec` functions in GHC Core requires looking at how GHC structures its Abstract Syntax Tree (AST) under the hood.

In standard Haskell, you can write a `let` block, and the compiler automatically figures out if the variables refer to each other. In Core, the compiler removes that ambiguity.

Here is exactly how `Rec` works, followed by the best documentation links for mastering GHC Core.

## How `Rec` Works in GHC Core

In the actual Haskell source code for the GHC compiler (specifically the `Core` module), a `let` expression is defined as containing a `Bind`. That `Bind` has exactly two constructors:

```haskell
data Bind b = NonRec b (Expr b)
            | Rec [(b, Expr b)]

```

This explicit separation exists for two critical reasons: **Compiler Pass Efficiency** and **Heap Allocation (Tying the Knot)**.

### 1. Compiler Pass Efficiency

Optimizer passes—like Dead Code Elimination, Strictness Analysis, and Inlining—need to know about dependencies.

If an optimizer sees `NonRec x expr`, it is a guarantee: `expr` **does not** contain `x`. The compiler can analyze `expr`, compute a property (like "is this strict?"), and linearly apply it to `x` without looking back.

If the optimizer sees `Rec [(x, expr1), (y, expr2)]`, it means `expr1` might call `y`, `expr2` might call `x`, or they might call themselves. To analyze a `Rec` block, the compiler cannot do a simple linear pass; it must compute a "fixed point" (iterating over the block until its analysis stops changing). By explicitly flagging `Rec`, GHC saves massive amounts of compile time because it only runs these expensive cyclic algorithms when strictly necessary.

### 2. Heap Allocation (Tying the Knot)

Operationally, `Rec` dictates how memory is allocated at runtime in the STG machine.

* **`NonRec`:** GHC allocates memory for the expression, then binds the pointer to the variable.
* **`Rec`:** GHC must perform a trick called "tying the knot." It first allocates "black holes" (empty placeholder closures) on the heap for all the variables in the `Rec` block. Then, it evaluates the expressions using the pointers to those black holes, and finally overwrites the black holes with the actual computed closures.

When you see a `Rec` block in your `.dump-simpl` output, you are seeing a cyclic graph of dependencies that will be instantiated in memory all at once.

### The Pretty Printer (PprCore)

GHC Core is an internal data structure running in memory. To output it to a text file using `-ddump-simpl`, GHC passes that AST through a module called `PprCore` (Pretty Print Core).

Because a mutually recursive block of functions can span hundreds of lines of Core output, the developers of GHC decided that a simple `let` syntax wasn't visually distinct enough. Here is how the in-memory AST maps to the text you are reading:

**In GHC's Memory (AST):**

```haskell
Let (Rec [ ("mine",   expr1)
         , ("$wmine", expr2) 
         ]) 
    (body)

```

**In the `.dump-simpl` text file:**

```haskell

mine = expr1

$wmine = expr2
end Rec }

```

### Why `end Rec }`?

The `end Rec }` string is entirely synthetic. It doesn't exist in the AST, and it isn't valid Haskell (or even valid GHC Core if you were trying to write Core by hand using a plugin).

It is injected by the pretty printer purely as a visual aid. When you are scrolling through 5,000 lines of deeply nested optimized STG/Core conversions, finding the exact end of a recursive loop without strict indentation rules is a nightmare. The `end Rec }` tag gives you a definitive, grep-able marker that the mutually recursive environment has ended and the main body of the expression is about to begin.

So, whenever you see `Rec { ... end Rec }`, you can mentally translate that directly to "this is a standard `Let` expression using the `Rec` constructor, and everything inside this block was stored as a list of pairs in the compiler."
